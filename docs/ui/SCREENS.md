│                                  │
└──────────────────────────────────┘

#### 狀態互動規範

1. **私密留言卡片 (Private Message Box)**:
   - **狀態為 Host / Approved**: 卡片解鎖，直接顯示後端回傳的私密留言明文（例如 LINE 群組連結、集合時間地點）。
   - **其他狀態**: 顯示鎖定圖示與提示文字「僅被批准的成員可見」。後端傳送的欄位值為空，保障資訊安全。
2. **底部動態按鈕 (Dynamic Status Button)**:
   - **未申請**: 顯示「申請加入」，點選後跳出對話框讓使用者填寫報名訊息，狀態轉為「審核中 (Pending)」。
   - **審核中**: 按鈕呈現停用或標示為「審核中」，可供使用者點選「取消申請」。
   - **已批准**: 按鈕顯示「已加入」或「退出揪團」。
   - **被拒絕**: 顯示「再次申請」按鈕。點選左側「查看拒絕原因」可開啟 Dialog 顯示 Host 填寫的 `rejection_reason`，點選「再次申請」可重新發送申請。
3. **下拉刷新**: 支持 Flutter `RefreshIndicator`，下拉時重新調用 `GroupEventCubit.refreshEvent(id)` 更新最新狀態。

### 16. 申請名單審核頁 (Review Applications Screen)

Host 用於管理與審核申請者名單的專屬畫面。

```text
┌──────────────────────────────────┐
│ ← 申請者審核               (4 待審)│
├──────────────────────────────────┤
│                                  │
│  🦊 林小花                       │
│     申請訊息：「百岳經驗 5 座，   │
│     體能良好，希望能加入！」     │
│     [ ✓ 批准 ]    [ ❌ 拒絕 ]      │
│     ───────────────────────      │
│                                  │
│  🐨 張三                         │
│     申請訊息：「想跟去拍照～」   │
│     [ ✓ 批准 ]    [ ❌ 拒絕 ]      │
│                                  │
└──────────────────────────────────┘
```

#### 審核互動流程

1. **點擊 [ ✓ 批准 ]**:
   - 觸發 API，狀態更新為 `approved`。
   - 後端自動在 `trip_members` 新增該使用者，完成行程的讀寫/唯讀共享。
   - 介面將該申請者移出待審名單，彈出成功 Toast。
2. **點擊 [ ❌ 拒絕 ]**:
   - 彈出「拒絕申請」對話框（AlertDialog）。
   - Host **必須**填寫拒絕原因（例如：「抱歉，此行程難度較高，建議先累積中級山經驗。」）。
   - 點擊對話框的確認後，觸發 API 將狀態更新為 `rejected`，並寫入 `rejection_reason`。
   - 該名單移出待審名單。
